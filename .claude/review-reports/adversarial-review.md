### `src/tspaint/output.py:311` — hard_segments no-anchor fallback picks the dominant state from an UNWEIGHTED per-segment sum, so the span-minority ancestry can be reported as the single hard tract.
*severity: silent-corruption · verified by: workflow (cached votes) · votes: 7 CONF / 0 REF*

- **Failure scenario:** Weak-signal haplotype (old admixture; high-level DEFAULT_DEADBAND=0.4) where no segment reaches the margin: one long informative segment [0,9e5) with posterior [0.55,0.45] (state 0) plus 30 short segments each 1e4 wide over [9e5,1.2e6) with [0.48,0.52] (state 1). anchors is empty, so state[:] = argmax(psum) with psum=[0.55+30*0.48, 0.45+30*0.52]=[14.95,16.05] -> argmax=1. hard_segments returns [(0, 1.2e6, 1)]: the whole haplotype is called B, though A covers 75% of the span. Verified by running the code. Downstream ancestry-proportion / dating (segments_as_frame, benchmark.score) silently records 100% of the wrong (minority) ancestry for every weak-signal haplotype.
- **Why it's wrong (spec/math):** The inline comment (line 309-310) states the intent is 'one tract of the most probable state overall'; 'overall' across the genome is a span (length) property, but psum (line 298,301) sums each merged segment's posterior once regardless of its (right-left) span, double-counting short trees relative to long ones. The main docstring (line 289) additionally promises 'raw per-locus argmax' here, a third, different behavior — so code, inline comment, and docstring all disagree. This regime (max prob <0.7 everywhere) is exactly the weak-signal case CLAUDE.md §9 documents (T_admix=1000: mean confidence 0.15), not merely degenerate.
- **Suggested fix:** Span-weight the fallback: initialise psum=0 and accumulate psum += (seg.right - seg.left) * p in the loop (line 301), so argmax(psum) is the genome-span-majority state; then reconcile the line-289 docstring to say 'single tract of the span-majority state' (drop 'raw per-locus argmax').

### `src/tspaint/benchmark/_msp.py:94` — parse_fb maps .fb posterior columns to tspaint states by sorted-pop-label position with no code_to_state remap, so any sample map whose integer population labels are not exactly {0..K-1} silently assigns RFMix/gnomix posteriors to the wrong states.
*severity: silent-corruption · verified by: hand (main-loop read) · votes: hand-verified*

- **Failure scenario:** User runs `tspaint benchmark rfmix` (or gnomix) with a sample map coding populations 1 and 2 (a mainstream convention). assign_states/export take these literally: K=3, truth states 1 and 2. RFMix's .fb columns are labelled :::1 and :::2. parse_fb does `pops=sorted(['1','2'])`=['1','2'] and treats column 0 as state 0, column 1 as state 1, padding state 2 with zeros. A query truly state 1 gets posterior [0.95,0.05,0.0] (argmax 0). Scored against truth state 1, every base mismatches -> the competitor is silently reported at ~chance balanced accuracy. - **Suggested fix:** Assign each .fb pop column to its actual state integer, not its sorted index: build the K-wide posterior by placing column for pop p at index int(p) when numeric (mirroring assign_states), else via a passed code_to_state / panel.state_label. ### `src/tspaint/pruning.py:198` — `loo` is returned as the bare outside/down message `U[c]`, which is the true leave-one-out marginal only for LEAF nodes; for an internal (non-leaf) labelled sample it omits the node's own descendant evidence, silently corrupting credibility and the introgression map for that node.
*severity: wrong-in-edge-case · verified by: workflow (cached votes) · votes: 8 CONF / 2 REF*

- **Failure scenario:** Tree: leaves 0,1 (both emit ~state A) are children of internal SAMPLE node 2, whose parent is latent root 3. Prune with pi=[.5,.5], slow Q. For node 2 the code returns loo[2]=[0.5,0.5] (just pi pushed one branch down from the root, ignoring children 0,1), while the true leave-one-out marginal excluding only node 2's own emission is [0.988,0.012] — a 0.49 error. In accumulate.py:143 `agree = res.loo[c][labels[c]]` (c = e.child, which may be this internal sample) then records agree=0.5 for a correctly-labelled internal reference, collapsing its learned credibility w_i toward the flagged-mislabel region; likewise introgression.py:167 `res.loo.get(s, pi)` yields a wrong `fit`/foreignness for any internal sample s (e.g. an ancient/ancestral reference). No crash, plausible-looking numbers.
- **Why it's wrong (spec/math):** CLAUDE.md §2.3/§3.3 define the LOO signal as 'what the REST of the tree says about this tip' with the node's OWN emission excluded — for an internal sample the 'rest of the tree' includes its descendants. The PruneResult.loo docstring (lines 55-57, 65-67) says 'leave-one-out marginal ... excluding the node's OWN emission', but the code excludes the entire subtree below the node, not just its emission. It coincides with the true LOO only because leaves have no subtree (`inside-without-own-emission` = flat), which is why the brute-force test suite (which never checks loo) misses it.
- **Suggested fix:** Do not equate loo with the outside message U. Compute loo[c] = normalize(U[c] * prod(msg[ch] for ch in tree.children(c))) — i.e. multiply the outside message by the product of c's children up-messages (the inside likelihood with c's own emission left out). For leaves the product is empty (ones), so loo[c] reduces to U[c] as before; for internal samples it now includes the descendant evidence. Equivalently loo[c] = normalize(U[c] * (Lnorm[c] / emission_c)) but prefer the child-message product to avoid division. Also fix loo[root] (currently pi) the same way for a sample root with descendants.

### `src/tspaint/parallel.py:484` — Parallel painting is NOT byte-identical to serial as the module claims: the segment-merge rule is anchored to the first segment of a run and is non-transitive under np.allclose, so re-anchoring at chunk seams makes the reported posterior depend on n_jobs whenever merge_tol exceeds the bit-identical floor.
*severity: wrong-in-edge-case · verified by: workflow (cached votes) · votes: 7 CONF / 0 REF*

- **Failure scenario:** Reproduced: simulate_admixture(Ne=1000,T_admix=30,T_split=5000,f_A=0.5), L=3e4, rho=5e-7, seed=3 -> 349 trees, Q=make_generator_2state(1e-3,1e-3). Comparing the full-genome posterior_table against _stitch_tracks of the per-chunk genome_chunks(ts, nch) tracks (exactly what posterior_table_parallel computes): at merge_tol=1e-2, nch=5, query node 0 at position 10065 the serial single loop reports posterior [0.8175,0.1825] while the parallel-stitched result reports [0.8024,0.1976] (|diff|=1.5e-2); the number and boundaries of segments also differ and change with the chunk count (nch=2..6). Root cause: a run of adjacent segments p0,p1,p2 where allclose(p0,p1) and allclose(p1,p2) hold but allclose(p0,p2) fails is split by serial (anchored to p0) as [p0+p1, p2], but a chunk boundary after p0 makes the worker re-anchor the run to p1, and _stitch_tracks then merges p0 with the p1-anchored segment, yielding [p0+p1+p2] as one segment with posterior p0. At the shipped default merge_tol=1e-12 this is essentially unreachable (posteriors merge only when bit-identical), but merge_tol is a public parameter on all three painters, and n_jobs defaults to all-CPUs/SLURM, so raising it makes the per-locus posterior silently machine-dependent.
- **Why it's wrong (spec/math):** The module docstring (lines 23-25) and the docstrings of posterior_table_parallel (lines 356-361) and loo_posterior_table_parallel (lines 430-435) assert the stitched result is 'byte-identical to serial (not merely allclose) for any worker count'. The serial reference merge in output.py _paint_tracks (lines 89-93) only updates segs[-1].right and keeps the run's FIRST posterior as the comparison anchor, an intrinsically non-transitive rule; _stitch_tracks (lines 483-486) applies the identical anchored rule at seams but over a run whose anchor was reset inside each worker (each worker starts with segs=[]). Exact equality with serial would require the merge to be either transitive or globally re-derived from the concatenated per-tree posteriors, not stitched from pre-merged chunk tracks.
- **Suggested fix:** Either weaken the claim in the docstrings to 'byte-identical only when adjacent merged posteriors are bit-identical (the default merge_tol=1e-12 regime); for larger merge_tol the segmentation is n_jobs-dependent', or make it truly n_jobs-invariant by stitching from UN-merged per-tree segments (return raw per-tree segments from workers and run one global merge pass on the parent), or re-run the merge over each seam's full run rather than only comparing dst[-1] to the incoming segment. The same fix applies to _stitch_foreignness_tracks (lines 492-508).

### `src/tspaint/pruning.py:198` — `loo` for an internal (non-leaf) labelled sample excludes its ENTIRE subtree, not just its own emission — contradicting the documented 'leave-one-out excluding the node's OWN emission' and silently corrupting credibility (w_i) and the introgression map for internal/ancient references.
*severity: wrong-in-edge-case · verified by: workflow (cached votes) · votes: 7 CONF / 0 REF*

- **Failure scenario:** Tree with an internal labelled reference: node 2 is a SAMPLE (label A) with descendant tips 0,1 and parent root 3; the descendants strongly support state B. The correct leave-one-out marginal (drop ONLY node 2's own emission, keep its descendant clade) is [0.315, 0.685] (favouring B) — verified by brute force. But the code sets loo = dict(U) where U[2] is only the parent-side outside message, giving loo[2] = [0.486, 0.514] (~50/50), because it discards the descendant clade 0,1. ts.samples() includes internal samples and build_emissions labels any sample in `labels`, so this reaches accumulate.py (agree = res.loo[c][label] = 0.486 instead of 0.315 -> w_i biased high, S_cred wrong) and introgression.py/loo_posterior_table (a ref's descendant-supported dissent never registers). Silent: no crash, just wrong numbers. Standard present-day tip references (the common case) are unaffected because for a tip the subtree IS just the emission.
- **Why it's wrong (spec/math):** PruneResult.loo docstring (lines 54-57, 65-67) and output.loo_posterior_table doc both define loo as 'the outside message ... excluding that sample's own emission' — for an internal sample 'the rest of the tree' includes its descendants, so descendants must be retained. The module explicitly advertises internal-sample support (docstring lines 6-7; ts_internal_sample test structure). CLAUDE.md §2.3/§3.3 define the LOO credibility as the label judged against the rest of the tree.
- **Suggested fix:** Store, in the up-pass, child_msg_product[u] = elementwise product of msg over u's children (=ones(K) for tips), then build loo[c] = normalize(U[c] * child_msg_product[c]) instead of dict(U). Confirmed to reproduce the brute-force LOO [0.315,0.685] for internal nodes and to be byte-identical to current output for tips (product = 1). Optionally do the same for the root (loo[root] = normalize(pi * child_msg_product[root])).

### `src/tspaint/archaic.py:483` — Ghost segments with a missing (nan) depth observation are unconditionally tagged INFORMATIVE, producing a confident-looking modern/ghost call over spans where no modern reference is reachable.
*severity: wrong-in-edge-case · verified by: workflow (cached votes) · votes: 3 CONF / 0 REF*

- **Failure scenario:** On a forested / multi-root inferred ARG (tsinfer/Relate/SINGER — the realistic input per §9/§7.4), a query haplotype can be isolated from EVERY modern reference over a span: for that tree interval every tree.mrca(s, r) is NULL, so _nearest_modern_depth returns nan and _depth_track stores depth=nan. Line 483 wraps this in Segment(..., INFORMATIVE) regardless. Verified: that gamma has top-two margin 0.8, which clears DEFAULT_DEADBAND=0.4, so hard_segments treats the no-data span as a CONFIRMED anchor and emits a confident 'modern' tract (or 'ghost', if learned pi0/burden is high) over a region with literally zero observations; the same nan span is also folded into burden (line 497), inflating/deflating the reported ghost burden. - **Suggested fix:** In the decode loop tag each segment by its observation: status = MISSING_INFO if np.isnan(obs[i]) else INFORMATIVE (pass it into Segment at line 483). ### `src/tspaint/validate.py:200` — mean_confidence uses |2·P(state)−1| (a 2-state formula on P(state 0)), which is wrong for K>=3 and inconsistent with track.py's confidence definition (max_p normalised out of [1/K,1]).

- **Failure scenario:** For a genuine K=3 painting (the spec's planned K-way / 3-way admixture; K is already threaded through painting_summary and balanced_accuracy), a locus with posterior [0.0, 0.5, 0.5] is maximally ambiguous between states 1 and 2. mean_confidence(state=0) returns |2*0.0-1| = 1.0 — a fully-confident reading of a 50/50 split. track.py:112 for the same posterior yields conf = (0.5 - 1/3)/(1 - 1/3) = 0.25 (correctly low). - **Why it's wrong (spec/math):** CLAUDE.md §2.1 requires the code to be generator-agnostic and generalise to {1..K} by a generator swap; §12 restates 'K-way by generator swap'. track.py documents and implements the K-correct confidence conf=(max_p−1/K)/(1−1/K) (track.py:66-67, L112). validate.mean_confidence hardcodes the 2-state |2P−1| on state 0, so the two callers define 'confidence' differently for K>=3 (they only coincide at K=2).
- **Suggested fix:** Compute confidence as (max(seg.posterior) − 1/K)/(1 − 1/K) (reading K from len(seg.posterior)), matching _Colorizer._alpha in track.py, instead of |2·P(state)−1|; keep the state param only for a signed/2-state read-out if needed.

### `src/tspaint/pruning.py:137` — The partial-likelihood product over ALL children (up-pass) and over sibling messages (down-pass cavity) is formed before a single normalization, so a wide polytomy (~>1000 near-balanced children) underflows to all-zeros: loglik -> -inf and posteriors silently fall back to the uniform 1/K.
*severity: robustness · verified by: workflow (cached votes) · votes: 5 CONF / 0 REF*

- **Failure scenario:** A marginal tree with a wide unresolved polytomy — a node with ~2000 sample children of comparable (near-balanced) emission, realistic at a deep unresolved coalescence in a large tsinfer/Relate ARG. In the up-pass L = emission * prod(msg[c] over all children) is computed first and normalized only afterwards (s=L.sum(); if s>0: L/=s), so L underflows to 0 before normalization; Zr=0, loglik=-inf (confirmed at k=2000; fine at k=500). In the down-pass, cavity = base * prefix[i] * suffix[i] is the product over ~k-1 sibling msgs and can likewise underflow to 0, whereupon U[c] (line 190), gamma[c] (line 193) and xi (line 196) are silently reset to the uniform 1/K fallback instead of the true posterior. Span-weighted loglik in accumulate.py becomes -inf, breaking any EM likelihood-based convergence monitoring, and the widest nodes' posteriors/xi can be silently uniform-ized (feeding wrong sufficient statistics for that edge).
- **Why it's wrong (spec/math):** The module docstring (lines 23-25) claims 'Likelihoods are normalised per node for numerical stability' — but per-node normalization only bounds cross-LEVEL growth (deep trees); it does not protect a single wide node whose child product underflows before the one normalization. CLAUDE.md §4.6/§12 make polytomies first-class ('internal nodes with >2 children exist', product over all children), and §8.7 flags numerics of large/stiff products as a validation item.
- **Suggested fix:** Renormalize incrementally INSIDE the child-product loops. Up-pass: after each `L = L * msg[c]`, rescale L by its sum and add log(sum) to cs. Down-pass: build the prefix/suffix sibling products with per-step renormalization (or normalize each msg once and carry its log-scale), keeping the running vector O(1) regardless of arity. Equivalently, work in log-space for the child product. This removes the -inf/uniform-fallback failure for wide polytomies while leaving posteriors unchanged for normal arities.

### `src/tspaint/bp/horizontal.py:45` — Docstring claims epsilon -> (K-1)/K is 'no coupling (recovers the per-segment input)', but A=(1-eps)I+(eps/K)J is only rank-1/uniform (truly decoupled) at eps=1, not at (K-1)/K.
*severity: minor · verified by: workflow (cached votes) · votes: 5 CONF / 0 REF*

- **Failure scenario:** A user reads the docstring and sets epsilon=(K-1)/K=0.5 (K=2) expecting bp_smooth to return the unsmoothed per-segment posteriors, but A=[[0.75,0.25],[0.25,0.75]] still couples neighbors, so an isolated flip is still partially smoothed and the output silently differs from the raw painting they intended.

- **Suggested fix:** Correct the docstring to state that eps=1 gives the uniform (no-coupling) transition, and that the actual per-breakpoint switch probability is eps*(K-1)/K (= eps/2 for K=2), so 'epsilon' is a switch-noise scale, not the switch probability itself.

### `src/tspaint/archaic.py:316` — The emission M-step is genome-span-weighted while the forward-backward E-step and the transition/pi0 M-steps are unweighted per-segment, so the reported loglik can be non-monotonic and the ghost emission fit mixes two inconsistent weighting schemes.
*severity: minor · verified by: workflow (cached votes) · votes: 2 CONF / 0 REF*

- **Failure scenario:** _forward_backward treats each tree-interval as ONE unweighted emission (B[t]^1), and acc_xi/acc_pi (lines 310-311) accumulate transitions/initials unweighted. But the ghost emission M-step (line 316: g = gamma[ok,k]*w with w=span) reweights the responsibilities by genomic span, so mu[1]/sd[1] are fit as a per-base-weighted MLE that does NOT maximize the Q-function the unweighted E-step gamma corresponds to. With a skewed segment-span distribution (a few very long segments vs many short ones — typical of an ARG), the span-weighted emission update can move the ghost mean/std away from the point the unweighted likelihood wants, so the loglik computed from the unweighted FB (float(np.log(c).sum())) can DECREASE across an iteration, contradicting the documented guarantee, and the early-stop test abs(ll-prev)<tol then fires on a non-monotone sequence.
- **Why it's wrong (spec/math):** GhostResult.loglik_history docstring (line 211) states the pooled Baum-Welch log-likelihood is 'non-decreasing' — the standard EM monotonicity property, which requires the M-step to maximize the same expected complete-data log-likelihood the E-step's responsibilities were computed under. Span-weighting only the emission sufficient statistics breaks that coupling. (A fully-consistent span-weighted HMM would raise B[t] to the power span[t] in the FB so the responsibilities and the M-step share one objective.)
- **Suggested fix:** Either (a) make the FB span-consistent — use B[t]**span[t] in _emission/_forward_backward and drop the ad-hoc *w in the M-step — or (b) drop the span weighting from the emission M-step (use g = gamma[ok,k]) so the unweighted FB and M-step agree. If the current hybrid is deliberate, soften the docstring's 'non-decreasing' claim and note the estimator is a per-base-weighted emission over per-breakpoint dynamics.

### `src/tspaint/cli.py:155` — CLI `paint --params --deadband X` silently ignores the supplied deadband and uses the value stored in params.npz.
*severity: minor · verified by: hand (main-loop read) · votes: hand-verified*
